Resumo executivo

A fundação crash-safe está pronta; falta o que torna isto "memória"

Em uma semana de M1, o EmbedMind ganhou uma engine de armazenamento completa — arquivo único, WAL, B-tree de registros, API pública — com uma rede de segurança de testes desproporcional ao tamanho do código: dois harnesses de crash e cinco alvos de fuzzing. O que ainda não existe é a única coisa que faz disto uma memória semântica: busca vetorial. É o próximo passo, e é o maior item técnico do M1.

Item 1.1 · concluído em commit anterior

O arquivo .mind: um formato binário paginado com WAL

A promessa central do produto é "um arquivo, nunca corrompe". Isso significa que antes de qualquer feature, a engine precisava de um formato de disco explícito — nada de structs serializadas cruas — com checksum em cada página e um log de escrita à frente (WAL) que sobrevive a kill -9 no pior momento possível.

Layout do arquivo principal

memory.mindpage_size = 4096 bytes (configurável)
página 0
Header — magic MINDFMT1, versão, page_size, ponteiros para a raiz da B-tree/HNSW/freelist, contador de transação, checksum xxh3 dos últimos 8 bytes.
páginas 1..N
B-tree de registros — folhas e nós internos com as memórias (item 1.2).
páginas 1..N
Overflow — continuação de registros grandes demais para uma página.
futuro
Vetores + HNSW + freelist — chegam no item 1.3.

O sidecar transitório

Enquanto há transações não consolidadas, existe um segundo arquivo — memory.mind-wal — que recebe as páginas modificadas antes de tocarem o arquivo principal. Ele só existe entre checkpoints; um store fechado normalmente é um único arquivo.

memory.mind-walapagado no fechamento limpo
header
magic MINDWAL1, versão, page_size, salt aleatório — impede que frames de uma geração antiga (truncada) sejam replayed por engano.
frame N
page_no · txn_id · flag commit · checksum — seguido da imagem completa da página. Cada frame carrega seu próprio checksum, seedado pelo salt.

Protocolo de commit: uma transação anexa todas as páginas modificadas ao WAL, chama fsync (FlushFileBuffers no Windows), e só então a última página carrega a flag de commit válida. A transação é durável se e somente se essa última verificação passar no disco.

Recuperação: toda abertura do arquivo varre o WAL do início, aplica só as transações com commit válido, e descarta qualquer cauda torta (torn write) — automático, silencioso, e coberto pelo harness abaixo.

Item 1.2 · concluído nesta sessão

A B-tree de registros e a API pública

Esta é a peça que transforma "um arquivo com checksums" em "uma loja de chave-valor de verdade". Chaves são ULIDs de 16 bytes — ordenados por tempo, então a ordem de leitura da árvore já é a linha do tempo, de graça. Valores são registros de memória codificados byte a byte.

API pública em uso

Esta é a superfície que embedmind-mcp e embedmind-cli vão chamar — nada abaixo dela é visível de fora do crate:

ChamadaO que fazGarantia
Store::remember(draft)Grava uma memória nova, gera o ULID e o timestamp.1 transação durável por chamada
Store::get(id)Busca por id; memórias esquecidas retornam None.idêntico a "não existe"
Store::forget(id)Marca tombstone (soft delete).forget redundante grava zero bytes
Store::iter() / iter_all()Itera em ordem de id = ordem de tempo.erro tipado em vez de pânico em arquivo corrupto

Uma decisão deliberada: não existe delete na B-tree. "Esquecer" é uma atualização que liga o bit de tombstone; o espaço — e qualquer cadeia de overflow associada — só é recuperado por um futuro embedmind vacuum. Simples, honesto, e documentado como dívida técnica aceita (ADR 0003), não como bug.

Item 1.8 · concluído nesta sessão

A rede de segurança: dois harnesses de crash e cinco fuzzers

A marca do EmbedMind é "nunca corrompe sua memória". Isso não é uma frase de marketing — é testado mecanicamente, e testado contra si mesmo: antes de confiar no harness, ele precisa provar que pega bugs reais.

O teste que prova o teste

Durante esta sessão, o fsync do caminho de commit foi removido de propósito, como experimento. Resultado: as três varreduras do harness de registros falharam instantaneamente, cada uma imprimindo a tupla exata que reproduz a falha — por exemplo:

I2/I3 violado (workload RememberHeavy, P 6, mode Before, seed 0xa100000006)

O fsync foi restaurado e as 65 execuções voltaram a passar. Esse tipo de verificação — quebrar de propósito para confirmar que o teste denuncia — é registrado no CHANGELOG por política de honestidade brutal do projeto.

O que cada crash simulado verifica

Os dois harnesses (tests/crash.rs na camada de páginas, tests/crash_records.rs na API pública) rodam a mesma varredura: para cada operação de I/O que modifica o disco, simulam uma queda de energia bem naquele ponto — e às vezes uma escrita "torta" (só parte da página chegou ao disco) — reabrem o arquivo e checam cinco invariantes:

I1

O arquivo sempre abre. Recuperação nunca falha em arquivo gerado pelo próprio harness.

I2

Toda transação que o programa viu como confirmada está inteiramente presente.

I3

Nenhum efeito de transação não confirmada aparece — sem meio-registro, sem ponteiro solto.

I4

Todo checksum de página no arquivo principal e no WAL sobrevivente é válido.

I5

O conteúdo lido bate exatamente com um modelo de referência em memória, alimentado só com as operações confirmadas.

Cinco alvos de fuzzing, cobertura em todas as plataformas

Como cargo-fuzz (baseado em libFuzzer) só roda em Linux, cada alvo é um corpo de função isolado em embedmind_core::fuzz — o mesmo código que o fuzzer ataca também roda como teste rápido e determinístico em cargo test, em qualquer plataforma, incluindo o Windows onde o founder desenvolve.

AlvoAtaca
fuzz_headerParser do header: magic, versões, tamanhos de página bizarros.
fuzz_pageFolhas, nós internos e cadeias de overflow da B-tree.
fuzz_recordDeserialização de MemoryRecord, incl. prefixos de tamanho hostis.
fuzz_wal_replayRecuperação completa com bytes arbitrários como sidecar do WAL.
fuzz_open_fullBytes arbitrários como um arquivo .mind inteiro — deve sempre retornar Ok ou erro tipado, nunca pânico.

O corpus inicial (arquivos-semente válidos, gerados pelo próprio codec) já está commitado em fuzz/corpus/. O CI roda uma passada curta (2 min/alvo) em todo PR e uma passada longa (1h/alvo) toda madrugada, acumulando cobertura ao longo do tempo.

Próximos passos

O que vem a seguir, e por que nessa ordem

A ordem não é arbitrária — vem do gráfico de dependências do próprio ROADMAP.md: nada de busca semântica sem vetor persistido; nada de MCP sem uma API estável embaixo dele.

1.3

HNSW paginado + embeddings ONNX embarcados

O maior item técnico do M1. Um índice HNSW próprio, persistido em páginas do mesmo arquivo — não uma biblioteca pronta que assume o grafo inteiro em RAM. Embeddings via all-MiniLM-L6-v2 quantizado (int8, ~23 MB, 384 dimensões), rodando 100% localmente via ONNX Runtime, sem chave de API.

Por que agora: é a peça que transforma um KV store em memória semântica — o motivo do produto existir. Também é a barreira técnica deliberadamente não terceirizada: bibliotecas prontas (hnsw_rs, usearch) exigem carregar o grafo inteiro na memória, o que quebra a promessa de abrir um arquivo de 1 GB sem travar. É "a moat", nas palavras do próprio DESIGN.md — por isso não é terceirizada.

1.4

Servidor MCP — remember / recall / forget

Casca fina em stdio JSON-RPC sobre a API que já existe. Zero lógica de domínio aqui — só parse, chamada, serialize.

Por que depois do 1.3: recall sem busca vetorial seria só um get disfarçado — não vale a pena expor via protocolo antes de ter algo semântico para devolver.

1.5–1.7

Contexto automático de projeto, instalação em 1 comando, README com benchmark

A camada de UX que faz o produto parecer mágico: o servidor infere o projeto pelo cwd do agente, filtra recall por escopo, e tudo isso é instalável com um único cargo install.

Por que por último: são os itens voltados para fora — precisam de algo funcionando embaixo para demonstrar. O benchmark honesto (mesmo perdendo em algo) é a peça que fecha o M1 e destrava o lançamento público do dia 35.
